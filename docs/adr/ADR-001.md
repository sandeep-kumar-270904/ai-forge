# ADR-001: Asynchronous PostgreSQL over Synchronous

*   **Status:** Accepted
*   **Date:** 2026-06-08

## Context
AIForge handles workflows that involve making HTTP requests to external LLM providers (OpenAI/Anthropic). These requests are highly latent, often taking 5 to 30 seconds. If we use a synchronous web framework (like Django or standard Flask) and a synchronous database driver (like `psycopg2`), the worker thread is blocked entirely while waiting for the network response.

## Decision
We will use **FastAPI** with `asyncio` and SQLAlchemy 2.0 configured with the **asyncpg** driver. All database operations (`session.execute`) must use the `await` keyword.

## Consequences
*   **Positive:** A single Uvicorn worker can handle thousands of concurrent LLM requests because it yields control of the event loop during network I/O.
*   **Negative:** Developers must strictly adhere to async programming paradigms. Accidental blocking calls (e.g., `time.sleep()`) will lock the entire event loop and freeze the application.

## Alternatives Considered
*   **Synchronous with Threadpools:** Using standard threaded Python (Gunicorn/WSGI). *Tradeoff:* Memory overhead per thread is too high for thousands of concurrent 30-second connections.

# ADR-003: Redis for Gateway Caching

*   **Status:** Accepted
*   **Date:** 2026-06-10

## Context
LLM API calls are expensive and slow. Many enterprise workflows involve repetitive tasks where the input prompt and parameters are identical (e.g., standard document summarization templates). We need a mechanism to prevent hitting the OpenAI API if we already know the answer.

## Decision
We will deploy **Redis** and implement a Semantic Caching layer in the Model Gateway. We will hash a combination of the `model_name`, `temperature`, and `prompt` text. If the hash exists in Redis, we return the cached response immediately.

## Consequences
*   **Positive:** Drastic reduction in API costs and latency (from ~5s to < 2ms) for repetitive queries.
*   **Negative:** Introduces a new infrastructure dependency. Cache invalidation logic must be managed.

## Alternatives Considered
*   **PostgreSQL Caching:** *Tradeoff:* Slower than Redis and unnecessarily burdens the primary database with transient key-value lookups.
*   **In-Memory Caching (LRU):** *Tradeoff:* Does not scale horizontally. If we have 10 API pods, each pod would have a disjointed, isolated cache, rendering it highly inefficient.
